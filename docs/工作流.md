# 在启智上 vibe coding：工作流与定位

**qzcli 是面向 coding agent 的启智（`qz.sii.edu.cn`）命令行工具——让在 tmux 里长期运行的 Claude 自己把「开发 → 调试 → 正式训练」这条循环全自动地跑起来。**

项目地址：<https://github.com/Sarfflow/qzcli>。它本身就是一个**自包含、即开即用的 Claude Code skill**——clone 进 skills 目录、`uv sync` 一次即可用。

## 背景：需求从 ssh 变成了「全自动循环」

上学期为了能在启智上方便地 vibe coding，我搓了个 rtunnel 来连接 Cursor。但它已经三个月没被打开过了。

原因是今年 coding agent 能力的阶跃式增长，彻底改变了 vibe coding 的形态：

- **以前**：靠 IDE + Tab 补全，人坐在编辑器前，刚需是把本地编辑器通过 ssh 连进容器。
- **现在**：一个常驻集群容器、在 tmux 里长期运行的 Claude。我只要打开网页输入指令就行。

刚需也随之转变——不再是「打通 ssh」，而是**打通整个开发-调试-正式训练的全自动循环迭代**。

## 痛点

直接在 GPU 实例里 vibe，有两个绕不开的问题：

1. **没有一台「既能上网又适合 vibe」的机器**。H 卡上不了网，没法在里面直接 vibe；CPU 实例没卡，也不好 vibe。结果只能挤在 4090 上。
2. **不符合用卡卫生**。在 4090 上 vibe 往往要占着实例很久，把宝贵的 GPU 资源耗在了写代码上。

## 推荐工作流

核心前提：**同项目共享 GPFS**，所以 CPU 实例、GPU 实例看到的文件内容完全一致——在哪写、在哪跑可以解耦。

1. **在 CPU 实例上开发代码，但不运行。** CPU 实例可联网、不占 GPU，适合 agent 长期常驻写代码。
2. **起 4090 实例配环境、跑 smoke test。** 因为共享 GPFS，GPU 实例直接就能看到刚写好的代码。调通后**保存为镜像**。
3. **用镜像起分布式训练，并监控训练数据。** 重计算任务交给分布式训练，CPU 实例上的 agent 负责盯着进展。

## 为什么这样更好

- **长期常驻 vs 按需占用**：CPU 实例可联网、不吃 GPU，agent 可以长期挂在上面写代码、监控训练任务；GPU 交互式建模只用来调试和配环境，**按需占用、agent 用完就及时关掉**。
- **好钢用在刀刃上**：真正的重计算全部交给分布式训练任务，GPU 资源不浪费在 vibe 上。
- **agent 比人更守纪律**：你需要睡觉，可能会忘记关实例；agent 不会。
